# 快速上手

這個 skill 是幫團隊把模糊反饋整理成可以發派的 Jira 任務。
適合 QA、PM、設計、測試與一般協作角色使用，不需要先懂 Jira API。

## 先做這一步：安裝 skill

第一次使用時，先依照 root README 的安裝方式，把 repo 放到 agent 的標準 skill 目錄：
- Claude Code：`~/.claude/skills/jira-intake`
- Codex：`${CODEX_HOME:-$HOME/.codex}/skills/jira-intake`

安裝完成後，重新開一個新 session，再開始下面流程。

## 你可以怎麼用

### 用法 1：先整理反饋
把以下任一種內容丟給 skill：
- 一段抱怨式描述
- 一張截圖
- 一段錄影
- 一個 Jira 草稿
- 一句很模糊的問題回報

skill 會幫你判斷：
- 這是 bug 還是需求
- 可不可以直接發派
- 是缺資訊，還是其實需要產品決策
- 還缺什麼資訊
- 如果原本就是 Jira 單，應該怎麼回寫 comment
- 建議怎麼寫成 Jira

### 用法 2：整理後直接建 Jira
草稿確認後，skill 可以直接建立 Jira 任務（需設定 Jira MCP 或 scripts）。

## 最簡單的操作流程

1. 把反饋丟給 Claude 或 Codex（文字 / 截圖 / 影片皆可）
2. skill 立即產出草稿，格式是標題、問題描述、優先序、負責人、標籤
3. 確認草稿內容是否正確
4. 若有看不懂的，直接回覆修改；若 OK，回「確認」
5. skill 詢問要建在哪個看板、sprint、指派給誰
6. 建立完成，回傳 Jira ticket 連結

## 什麼情況不要直接建單

以下情況先不要急著發派：
- 只有影片，沒有文字說明
- 只有影片，而且沒標出哪一秒有問題
- 看不出是哪個頁面或元件
- 不清楚是 bug 還是需求
- 不清楚預期行為是什麼
- 預期行為沒有任何依據，例如沒有 Figma、PRD、或 owner 確認
- 標題和描述互相矛盾

## 一句話理解

這不是幫你亂猜需求的工具。
這是幫你把反饋先整理乾淨，再安全地送進 Jira 的工具。
