# Dispatch Checklist

The default is to draft a ticket. Use this checklist only to decide whether to stop and ask one follow-up question instead.

## Draft immediately when

- The page or feature is identifiable
- The issue (actual behavior) is clear enough to describe
- The fix direction is reasonably obvious, even if details are missing

Missing app version, build number, or exact repro frequency does not block a draft.

## Stop and ask one question when

Only one of these is true and it makes the fix direction non-obvious:

- Cannot identify the page, feature, or component at all
- Cannot tell what is wrong (no actual behavior described)
- The expected behavior is a personal preference with no spec, and the correct fix is genuinely unclear
- The report contains multiple unrelated issues that cannot be cleanly separated into one ticket

## Flag inline (do not stop the flow)

Include a note in the draft when:

- The fix is clear but source of truth is unconfirmed (e.g. "建議確認 Figma 設計稿再動工")
- The report may overlap with an existing ticket (e.g. "建議先確認是否有重複單")
- The expected behavior depends on a product decision not yet made

## Ask only one question

If a follow-up is necessary, ask only the single most blocking question.
Do not list multiple questions. Do not explain why you are asking.
